# build-deck v1.1 — Motion Design Doc

**Date:** 2026-05-19
**Status:** Awaiting approval
**Target release:** v1.1
**Scope:** `claude-skills/build-deck` skill (template only)

## Goal

Add motion to Present mode so slide transitions feel intentional, slides "land" with a subtle entrance animation, and each theme has a distinct background-motion signature. All purely CSS, no new JS libraries, fully gated by `prefers-reduced-motion`.

## Motivation

v1.0.0 themes differ in palette, typography, and slide chrome — but Present mode is silent. Pressing `→` instantly swaps slides with no transition; nothing on the page moves. For pitch delivery and walk-through review, motion adds rhythm and emphasis. It also gives themes another axis to be distinct on.

## Non-goals

- No JavaScript animation libraries (GSAP, Motion One, three.js). Pure CSS + Web Animations API where needed for sequencing.
- No per-element user-authored animation overrides (e.g., "make this card pulse"). Themes pick a default; users don't fine-tune per slide.
- No animated slide builds / progressive reveals within a single slide (click → reveal next bullet). Whole slide enters as a unit.
- No motion on the topbar, dropdown, or navigation chrome — only on slides and slide content.
- No three.js / WebGL. Reserved for future ("Galaxy" showcase theme), not v1.1.

## Design

The motion system has three independent layers. Each can be enabled or disabled in isolation; they compose cleanly.

### Layer A — Slide transitions (between slides in Present mode)

When `showSlide(newIdx)` is called and `newIdx !== currentIndex`, animate the swap.

**Mechanism.** In Present mode, slides are stacked absolutely. The outgoing slide gets `.is-exiting` and the incoming slide gets `.is-entering` for the animation duration; both classes are cleared on `animationend`. The `.is-current` class moves immediately so chrome (page numbers, progress) reflects the new position.

**Transition variants** — picked by `data-transition` attribute on `<html>`:

| `data-transition` | Behavior | Duration |
|---|---|---|
| `fade` | opacity 0→1 in, 1→0 out, simultaneous | 320ms in, 280ms out |
| `slide-h` | translateX(40px → 0) in, (0 → -40px) out, opacity cross-fade | 380ms in, 320ms out |
| `zoom` | scale(.96 → 1) + opacity in, scale(1 → 1.04) + opacity out | 420ms in, 380ms out |
| `cut` | no animation | 0 |

**Per-theme defaults** when `data-transition` is `auto` (the default):

| Theme | Default | Why |
|---|---|---|
| Terracotta | `fade` | warm, classical |
| Carbon | `slide-h` | technical, deliberate |
| Berry | `fade` | editorial softness |
| Lab | `slide-h` | clean, predictable |
| Mono | `slide-h` | code-tab feel |
| Aurora | `zoom` | atmospheric |
| Brutalist | `cut` | no motion is the statement |
| Dusk | `fade` | premium, soft |

**User override.** A new "Motion" selector in the topbar (visible only in Present mode) lets the user pick from Auto / Fade / Slide / Zoom / Cut. Persisted to `localStorage.deck-transition`. Default `auto` = theme's pick.

### Layer B — Element entrance cascade

When a slide becomes `.is-current`, its children stagger-in over ~500ms.

**Targets**: `.pillar`, `.c-card`, `.flow__card`, `.q-card`, `.phase`, `.arch-card`. Also `.slide__title`, `.slide__subtitle`, `.title-slide__main`, `.title-slide__sub`.

**Choreography**:

- Title elements: enter at 0ms (alongside the slide transition).
- Subtitle: 80ms after title.
- Cards / pillars / nodes: starting at 160ms, 60ms stagger per item (n=3 cards → 160 / 220 / 280ms).
- Footer / objective bar: 320ms.

**Animation**: `translateY(12px) + opacity 0` → `translateY(0) + opacity 1`, ~480ms, `ease-out`. Universal across themes — no per-theme variation here, the cascade itself is the consistent rhythm.

**Implementation**: CSS `@keyframes` named `enter-up`, applied via `.slide.is-current .pillar { animation: enter-up .48s var(--ease-out) both; }` with per-child `animation-delay` via `:nth-child()`.

### Layer C — Per-theme background motion

Each theme has a distinct background-motion signature in Present mode. Some themes are explicitly static — that's the statement.

| Theme | Motion | Implementation |
|---|---|---|
| **Aurora** | Gradient hue-drift, 20s loop | `background-size: 200% 200%`, animate `background-position` |
| **Mono** | Dotted grid pulse on each slide change | Pseudo-element on slide, animates opacity + scale once per `.is-current` change |
| **Carbon** | Scanline drift, ~8s loop | Pseudo-element with `repeating-linear-gradient` of horizontal lines, animate `background-position-y` |
| **Berry** | Vignette breathe, 8s loop | Pseudo-element with radial-gradient, animate opacity .5 ↔ 1 |
| **Terracotta** | Static paper grain (no motion, but a fixed SVG noise texture for warmth) | Pseudo-element with inline SVG noise pattern |
| **Dusk** | None in v1.1 (deferred to v1.2 — particle field needs JS) | — |
| **Brutalist** | **Strict static** (no motion is the statement) | — |
| **Lab** | **Strict static** (clinical minimalism) | — |

All Layer C motion is fixed-position behind the slide content, low-opacity, and `pointer-events: none`. It plays only in Present mode (no motion in Review).

### Reduced-motion safety

All Layer A/B/C animations live inside:

```css
@media (prefers-reduced-motion: no-preference) { /* animations here */ }
```

…AND an explicit kill-switch:

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
  }
}
```

Users who set "Reduce Motion" in OS preferences get instant slide swaps, no entrance cascade, no background loops. The deck remains fully functional and accessible.

### State

| Key | Values | Default |
|---|---|---|
| `localStorage.deck-transition` | `auto`, `fade`, `slide-h`, `zoom`, `cut` | `auto` |

When `auto`, JS sets `data-transition` on `<html>` to match the active theme's default; on theme change, the attribute updates if user hasn't picked an override.

## Risks & open questions

- **Stacked positioning in Present mode.** Layer A requires slides to be absolutely positioned in Present mode (so old + new can both be visible during transition). The current layout sets `display: none` / `display: block`. Switching to absolute stacking is a non-trivial CSS change and could regress slide content layout. Mitigation: scope the absolute positioning to `html[data-mode="present"]` only; Review mode unchanged.
- **PDF export.** Print output must not include animations. `@media print` already hides chrome; animation kill-switch handled by `prefers-reduced-motion: reduce` from the print pipeline (Chromium honors this when printing).
- **Carbon scanline aesthetic.** Risk of looking gimmicky or interfering with text legibility. Acceptance: opacity ≤ 1.5%, only visible against solid card backgrounds. Walk every slide type before merge.
- **Aurora gradient performance.** Animating `background-position` on a fullscreen gradient runs on the compositor — should be smooth. Verify FPS in Chrome DevTools Performance tab before merging.
- **Berry vignette legibility.** A breathing dark vignette could pull attention away from content. Acceptance: max opacity .08; visible but not distracting.

## Acceptance criteria

1. Pressing → in Present mode plays the selected transition; old slide exits and new slide enters with no visible gap or flicker.
2. Element entrance cascade visible on first show of each slide — cards/pillars stagger in over ~500ms total.
3. Transition selector in topbar (Present mode only) cycles Auto / Fade / Slide / Zoom / Cut; selection persists across reload.
4. When `data-transition="auto"`, switching themes updates the effective transition to the new theme's default.
5. Aurora's gradient visibly drifts in Present mode over a 20s loop; static in Review mode.
6. Mono's dotted-grid pulse fires each time a slide becomes current.
7. Carbon's scanline visible and drifting; legibility unaffected (manual walk-through).
8. Berry's vignette breathes 8s loop; max opacity ≤ .08.
9. Terracotta has a fixed paper-grain texture (no animation, but visible as warmth).
10. Brutalist and Lab are explicitly static in Present mode — no slide-bg motion, no entrance cascade detectable to the eye (verify by DevTools timeline).
11. Setting OS-level "Reduce Motion" disables all animations; slides still swap instantly and remain navigable.
12. PDF export from Present mode produces visually identical output to v1.0.0 (no animation artifacts in print).
13. All 8 themes render every slide type without overflow or broken layout after the motion pass.
